# AI × Web3 最小交叉流程图 — 治理助手（示例 ③）

> 场景：DAO 治理助手 → 提案总结 → 人工审核 → 公开记录
> 把长提案整理成可理解的摘要，辅助成员决策

---

                         ┌──────────────┐

## 六要素标注

| # | 要素 | 对应节点 | 说明 |
|---|------|---------|------|
| ① | **谁发起任务** | 👤 DAO 成员 | 成员让治理助手帮忙分析提案 |
| ② | **谁执行** | **AI**: ①-④, ⑨ — 抓取、摘要、验证、归档<br>**人**: ⑤, ⑧ — 审核、验证<br>**链**: ⑦ — 链上执行投票 | |
| ③ | **哪步需要钱包签名/付款/授权** | 🔑 节点 ⑦ 链上投票 | 投票需要连接钱包、确认 Gas、签名交易 |
| ④ | **哪步必须人工确认** | 👤 节点 ⑤ 人工审核 | AI 的提案摘要和风险标记必须由人审阅后再投票 |
| ⑤ | **如何验证结果** | 👤 节点 ⑧ 验证记录 | 通过 Snapshot 或链上浏览器确认投票已成功记录 |
| ⑥ | **可能的风险点** | 各节点 ⚠️ 标注 | 链下数据源不可用、LLM 遗漏细节、虚假引用、签名非预期提案 |

## 三条边界

```
┌──────────────────────────────────┐
│  AI 层（节点 ①-④, ⑨）          │
│  拉取提案 → 摘要 → 链上验证 → 归档 │
├──────────────────────────────────┤
│  人层（节点 ⑤, ⑧）              │
│  审核摘要 → 确认 → 验证链上记录    │
├──────────────────────────────────┤
│  链上/签名层（节点 ⑥a, ⑦）       │
│  投票签名 → 链上执行 → 永久记录    │
└──────────────────────────────────┘
```

---

## 和示例 ① / ② 的核心差异

| 对比项 | 示例 ③ 治理助手 |
|--------|---------------|
| **链上操作** | 相对简单——只需一笔投票签名 |
| **AI 的核心价值** | 不是执行，而是**信息压缩和风险揭示** |
| **主要风险** | 不是合约漏洞，而是**信息遗漏或偏差** |
| **验证方式** | 链上记录 + 公开可见 |
| **链下依赖** | 依赖 Discourse/Snapshot 等链下数据源 |
