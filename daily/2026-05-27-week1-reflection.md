## 1. 一个我重新理解的 AI 概念：Agent Workflow & Tool Use

### 之前以为
Agent = 一个能对话的 AI，问它什么答什么，最多能联网搜索。

### 重新理解

Agent 的核心不是"能聊天"，而是**能行动**。它有一套完整的「感知 → 推理 → 行动 → 反馈」循环：

```
用户输入
  ↓
Agent 解析意图（LLM 推理）
  ↓
决定需要什么工具（搜索 / 读文件 / 调 API / 发交易草稿）
  ↓
调用工具，拿到结果
  ↓
LLM 分析结果，决定下一步
  ↓
循环直到任务完成 → 输出给用户
```

对我的启发是：

1. **Tool Use 是 Agent 和普通 LLM 的分界线。** Hermes 能读写文件、查数据库、发消息，这不是"功能"，这是 agent 的架构能力。没有工具的 LLM 只是一个知识回顾器。

2. **Workflow 不是提示词写得好，而是步骤设计得好。** 我在设计 DeFi 兑换助手时深有体会：意图解析 → 查询 → 生成草稿 → 模拟 → 人类确认 → 验证，每一步都是独立的逻辑单元，不能跳，不能合并。

3. **安全边界要设计在架构层面，不能依赖提示词。** 我那个设计的"三条铁律"（永不接触密钥、永不自动签名、永不绕过确认）是在工作流里硬编码的断点，不是靠 LLM"自觉遵守"。

4. **Hermes 的 "AI 最小实践" 让我看到了 Agent 的完整能力边界：** 交易解释器（只读）、风险摘要（文本生成）、钱包授权检查（链上查询）、协议文档 RAG（检索）、DAO 提案 Agent（工具链编排），这些不是 demo，是 agent 真正的生产场景。

### 实践关联

我在设计 DeFi Swap Agent 时，把 workflow 拆成了 4 个 Stage + 3 道确认墙。Agent 在每个 Stage 做什么、不做什么，是通过阶段边界定义的，不是靠 LLM 自己判断。这比"写好 prompt 让 AI 自觉"安全得多。

## 2. 一个我重新理解的 Web3 概念：钱包 & 签名

### 之前以为
钱包 ≈ 装币的 App，和支付宝差不多。私钥 ≈ 密码。

### 重新理解

钱包最核心的能力不是存钱，而是**签名**。

```
以太坊上的 "账户" 不是邮箱账号，而是一个算法产出：
  私钥 → 公钥 → 地址（不可逆）

"登录" 不是输入密码 + 验证码，而是：
  签名一段消息 → 网络验证签名来自持有私钥的人

"转账" 不是"从账户A划扣100到账户B"，而是：
  构造交易 → 用私钥签名 → 广播到网络 → 矿工/验证者执行
```

对我冲击最大的几个认知：

#### a) EOA ≠ 智能账户 ≠ 多签

| 类型 | 控制者 | 特点 |
|------|--------|------|
| EOA | 单一私钥 | 最基础，丢私钥就丢资产 |
| 智能账户 | 合约逻辑 | 可编程（社交恢复、限额、自动执行） |
| 多签 | N 个签名者中 M 个签名 | 集体决策，单点故障小 |

我之前以为"EOA 就是钱包"，但智能账户才是 Agent 真正需要的——Agent 可以拥有一个**受约束的智能账户**（有每日限额、白名单地址），而不需要持有完整的私钥。

#### b) 签名不只有交易签名

- **交易签名**（EOA 签交易 → 广播上链）→ 花 Gas，改变链上状态
- **消息签名**（EIP-712 结构化消息）→ 离线免费，用于 Permit2 等场景
- **消息验签** → 验证身份，不涉及资产

这在设计 DeFi Agent 时很关键：Permit2 的链下签名是「消息签名」，所以 Agent 可以安全地生成 EIP-712 签名草稿，真正广播的交易留给用户。

#### c) Gas 不是"手续费"，是计算资源定价

EIP-1559 把 Gas 拆成了 Base Fee（网络自动调节）+ Priority Fee（给小费抢位置）。L2 省 gas 不是因为"便宜"，而是因为计算外包给 L2 排序器，只在 L1 发布压缩后的数据。

### 实践关联

我在设计 DeFi Agent 的"三条铁律"时，核心就是**Agent 绝不接触私钥，绝不自动签名**。钱包是人类的专属工具，Agent 只生成 calldata 和签名草稿，签名这一步必须在人类控制的设备上完成。

## 3. 一个 AI × Web3 交叉问题：Agent 可以发起支付吗？

这个问题我在设计 DeFi Swap 工作流时反复思考过。

### 我的结论

**Agent 可以规划支付、构造交易数据、发起支付请求，但绝不应该自动签名并广播。** 这个边界画在签名这里。

### 具体分析

| 场景 | Agent 能做什么 | Agent 不能做什么 |
|------|---------------|-----------------|
| Agent 发起支付给服务商 | 构造交易 calldata，生成请求 | 自动签名并广播 |
| Agent 自动续费订阅 | 检测到期，生成支付请求推送用户确认 | 自动扣款 |
| Agent 支付 Gas | 使用预设的 Gas 站合约 | 持有私钥直接签名 |
| Agent 在 DAO 中执行提案 | 构造执行数据 + 模拟 | 自动执行（需多签确认） |

**为什么画在这里？**

因为签名等价于授权。一旦 Agent 能自动签名，就等于 Agent 拥有了私钥的完整控制权。那么：
- 如果 Agent 被 prompt injection 攻击 → 攻击者可以控制签名 → 转走资产
- 如果 Agent 的模型输出有幻觉 → 可能签错地址/金额
- 中间人篡改 calldata → 用户都没机会发现

### 安全可行的方案

一种折中方案是**受约束的智能账户**：

```
智能账户规则：
  每天转账上限：100 USDT
  白名单地址：仅限特定合约
  需要多签确认：超过 10 USDT 的操作
```

Agent 触发智能账户的 `execute()` 函数，但账户合约本身强制执行这些限制——不需要信任 Agent 的正确性，只信任合约规则。这比"信任 AI 不会出错"靠谱得多。

### 我的实践

在设计 DeFi Swap Agent 时，我设了三道确认墙：
1. 合约地址核对（人类从官方来源交叉验证）
2. Permit / Approve 参数核对（钱包内逐字核对）
3. Swap 参数 + calldata 核对（签名前再三确认）

这三道墙就是我对这个问题的答案：**Agent 可以无限接近签名，但签名本身必须是人的最后一道防线。**

## 5. 一个我还没解决的问题 / 下一周想继续探索的方向

### 还没解决的问题：Agent 的链上身份怎么建立？

目前的设计里，Agent 做了所有"非签名"的工作——查询、构造、模拟、提醒。但 Agent 的身份仍然绑定在**用户的账户**上：

- Agent 查 allowance → 用的是用户地址
- Agent 生成 calldata → to 和 from 都是用户的数据
- Agent 无法独立持有资产、无法独立授权

**我的困惑：**

如果你想做一个**真正自治的 Agent**（比如一个自动做市的 Agent，或一个自动管理资产生息的 Agent），它需要：

1. 一个链上身份（地址）
2. 一定的资产控制权（有限度的）
3. 能自己付 Gas

这和"人类必须签名"有根本矛盾。目前的思路：

- **智能账户 + 规则约束**：Agent 控制一个智能账户，但账户规则限制了它能做什么（限额、白名单、多签）
- **代付 Gas 网络**：通过 ERC-4337 的 Paymaster，Agent 不需要持有 ETH 付 Gas
- **Session Key**：短期授权的密钥，Agent 在 session 有效期内可执行有限操作

这些方向我都想进一步研究，尤其是 Session Key + 智能账户的组合。
