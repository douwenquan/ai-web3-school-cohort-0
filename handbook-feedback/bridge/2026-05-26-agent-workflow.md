# AI × Web3 Bridge 学习笔记：Agent Workflow（Agent 工作流）

> 基于 Handbook AI × Web3 Bridge 章节  
> 难点不是让 AI 帮用户操作，而是把日常操作拆分成可以验证的步骤

---

## 0. 核心原则

AI × Web3 的难点不是让 AI 帮助用户执行操作，而是：

```
将操作拆解为可验证的步骤
  ↓
寻找其中可以由 AI 替代的步骤
  ↓
识别哪些高风险环节仍需人工确认
```

### Agent Workflow 的核心组件

| # | 组件 | 说明 |
|---|------|------|
| 1 | **Task Graph** | 把目标拆成节点和依赖 |
| 2 | **State Machine** | 执行过程有明确的状态 |
| 3 | **Human in the Loop** | 关键风险点由人决断 |
| 4 | **Retry & Fallback** | 异常场景的处理策略 |
| 5 | **Trace** | 全链路追踪与复盘 |
| 6 | **Evaluation Harness** | 系统化测试框架 |
| 7 | **Regression Set** | 防止安全性退化 |

---

## 1. Task Graph（任务图）

### 核心定义

**Task Graph** 把目标拆成节点和依赖，**不是让 Agent 一口气自由执行**——而是每一步都有清晰的输入、输出和停止条件。

### 示例：低风险 Swap 的 Task Graph

```
用户目标: "把 100 USDC 换成 ETH"

Task Graph 拆解:
                                    ┌─────────────┐
                                    │ 读取用户目标  │
                                    │ 和限制条件    │
                                    └──────┬──────┘
                                           ▼
                                    ┌─────────────┐
                                    │ 获取余额和    │
                                    │ Allowance    │
                                    └──────┬──────┘
                                           ▼
                                    ┌─────────────┐
                                    │ 查询价格和    │
                                    │ 流动性       │
                                    └──────┬──────┘
                                           ▼
                                    ┌─────────────┐
                                    │ 生成候选交易  │
                                    └──────┬──────┘
                                           ▼
                                    ┌─────────────┐
                                    │ 模拟交易     │
                                    └──────┬──────┘
                                           ▼
                                    ┌─────────────┐
                                    │ 展示风险     │ ← Human in the Loop
                                    └──────┬──────┘
                                           ▼
                                    ┌─────────────┐
                                    │ 用户确认     │ ← 必须人工
                                    └──────┬──────┘
                                           ▼
                                    ┌─────────────┐
                                    │ 发送交易     │
                                    └──────┬──────┘
                                           ▼
                                    ┌─────────────┐
                                    │ 记录最终结果  │
                                    └─────────────┘
```

### 每一步的配置

```
每一步都可以设置:
  - 输入（依赖上一步的哪些输出）
  - 输出（传递给下一步的数据）
  - 权限（允许调用哪些工具）
  - 停止条件（失败后是重试、跳过还是中断）

好处:
  - Agent 可追踪执行
  - 问题发生时知道哪一步出了问题
  - 可以回放到某一步重新执行
```

## 2. State Machine（状态机）

### 核心定义

**State Machine** 让 Agent 执行过程有**明确的状态**，不会出现 "执行到哪了" 的模糊状态。

### 常见的状态

```
                    ┌─────────────┐
                    │   Draft     │ ← 初始草稿
                    └──────┬──────┘
                           ▼
                    ┌─────────────┐
                    │ Context     │ ← 上下文加载完成
                    │ Loaded      │
                    └──────┬──────┘
                           ▼
                    ┌─────────────┐
                    │ Plan Ready  │ ← 计划已生成
                    └──────┬──────┘
                           ▼
                    ┌─────────────┐
                    │ Simulation  │ ← 模拟已完成
                    │ Filled      │
                    └──────┬──────┘
                           ▼
               ┌───────────┴───────────┐
               ▼                       ▼
        ┌──────────────┐      ┌──────────────┐
        │ Waiting User │      │   Auto       │
        │ Confirmation │      │   Execute    │
        └──────┬───────┘      └──────┬───────┘
               ▼                     ▼
        ┌──────────────┐      ┌──────────────┐
        │  Submitted   │──────│  Confirmed   │
        └──────────────┘      └──────┬───────┘
                                     ▼
                            ┌────────────────┐
                            │  Reverted /    │
                            │  Cancelled     │
                            └────────────────┘
```

## 3. Human in the Loop（人在回路）

### 核心定义

**把人放在关键风险点**——在高风险交易、超出政策等情况下，由人类进行最终的决断。

### 什么情况需要 Human in the Loop

| 场景 | 风险等级 | 是否需要人 |
|------|---------|-----------|
| 读取余额 | 🟢 低 | ❌ 无需 |
| 查询价格 | 🟢 低 | ❌ 无需 |
| Swap ≤ 100 USDC | 🟡 中 | 可自动（有额度限制） |
| Swap > 1000 USDC | 🔴 高 | ✅ 必须 |
| 首次授权 (Approve) | 🔴 高 | ✅ 必须 |
| 合约升级 | 🔴 最高 | ✅ 必须 |
| 未知合约交互 | 🔴 高 | ✅ 必须 |

### 设计原则

```
不是问 "Agent 能做什么"，
而是问 "什么必须由人来做"。

把 Agent 的自动化能力和人的判断力结合起来。
```

## 4. Retry & Fallback（重试与降级）

### 核心定义

处理工具失败、网络异常、模型输出不合格、交易状态不确定等场景。

### ⚠️ 不要盲目重试

```
✅ 可以重试:
  读取余额失败 → 网络波动 → 重新请求

❌ 不能盲目重试:
  交易发送失败 → 需要先判断是否已经广播了
  如果已经广播了但还没确认 → 应该查询状态，不是重新发送
```

### 常见场景的处理策略

| 场景 | 处理方式 |
|------|---------|
| **RPC 节点超时** | 切换到备用节点重试 |
| **Gas 估算失败** | 提高滑点容忍度或降低复杂度 |
| **交易确认超时** | 查询交易状态，判断是否已广播 |
| **模型输出格式错误** | 重新请求（带更严格的格式要求） |
| **合约调用 revert** | 分析 revert 原因，调整参数重试或放弃 |

## 5. Trace（全链路追踪）

### 核心定义

**Trace** 跟踪每一步的输入、判断、工具调用和结果，并进行记录。有助于复盘整个操作流程的问题点。

### 追踪记录的内容

```
Step 1: 读取用户目标
  Input:  用户消息 "把 100 USDC 换成 ETH"
  Output: { target: "swap", tokenIn: "USDC", tokenOut: "ETH", amount: 100 }
  Tool:   无
  Status: ✅

Step 2: 获取余额
  Input:  { token: "USDC", address: "0xuser..." }
  Output: { balance: "500 USDC", allowance: "0" }
  Tool:   RPC.balanceOf, RPC.allowance
  Status: ✅

Step 3: 模拟交易
  Input:  { from: USDC, to: ETH, amount: 100 }
  Output: { estimatedOutput: "0.05 ETH", slippage: "0.3%", gas: "$2.50" }
  Tool:   Simulation API
  Status: ✅

...
```

### Trace 的价值

```
调试时:  快速定位问题出在哪一步
审计时:  完整记录 Agent 的每一步决策
改进时:  分析哪些步骤可以优化或缩短
```

## 6. Evaluation Harness（评估框架）

### 核心定义

**Evaluation Harness** 用来测试 Agent 在不同任务、风险和异常场景下的表现。

### 对链上 Agent 的评估标准

```
❌ 不只看: "回答得好不好"

✅ 还需要看:
  - 是否正确识别越权请求
  - 是否识别了错误的链
  - 是否识别了错误的合约地址
  - 是否提示了用户需要关注的风险
  - 是否在不确定时选择了保守策略
```

### 测试场景示例

| 测试类型 | 场景 | 期望行为 |
|---------|------|---------|
| 正常操作 | 用户想 swap 100 USDC | 正常执行 |
| 越权请求 | 用户要求 transfer 全部资金 | 拒绝 + 提示风险 |
| 错误链 | 用户给的合约地址不在当前链上 | 检测到链不匹配 + 警告 |
| 高滑点 | 池子流动性不足，滑点过高 | 提示风险 + 需要用户确认 |
| 未知合约 | 用户想交互一个没有审计的合约 | 提示未知风险 + 强制人肉确认 |

## 7. Regression Set（回归测试集）

### 核心定义

**Regression Set** 是一组固定测试用例，用来防止模型、Prompt、工具或策略更新后**安全性的退化**。

### 使用方式

```
每次更改模型或工具前:
  1. 跑一遍 Regression Set
  2. 对比上次的结果
  3. 确认 Agent 没有退化
  4. 如果有退化 → 回滚或修复后再上线
```

### 回归测试应该覆盖

| 类别 | 用例数 | 示例 |
|------|--------|------|
| 正常需求 | 10 | 常规 Swap、查询、转账 |
| 边界情况 | 10 | 金额为 0、极小额、极大额 |
| 已知 Bug | 5 | 之前修过的漏洞场景 |
| 恶意/注入 | 5 | Prompt Injection、越权请求 |

## Agent Workflow 全景总结

```
用户意图
    │
    ▼
┌────────────────────────────────────────────┐
│           Task Graph 分解                   │
│  目标 → 节点1 → 节点2 → ... → 执行完成      │
└────────────────────────────────────────────┘
    │
    ▼
┌────────────────────────────────────────────┐
│           State Machine 状态流转             │
│  Draft → Loaded → Ready → Exec → Done/Fail │
└────────────────────────────────────────────┘
    │
    ├── Human in the Loop ← 关键风险由人决断
    ├── Retry & Fallback ← 异常处理不盲目
    ├── Trace ← 全链路可追踪
    │
    ▼
┌────────────────────────────────────────────┐
│          Evaluation / Regression            │
│  每次变更前跑测试集，防止安全性退化          │
└────────────────────────────────────────────┘
```
