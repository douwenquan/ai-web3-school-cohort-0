## 0. Oracle 是什么

### 核心定义

Oracle（预言机）是**链上系统和外部世界之间的桥**。区块链本身无法主动获取外部数据，Oracle 负责把链外数据带入链上，让智能合约可以基于现实世界的信息执行。

### 哪些协议依赖 Oracle

| 协议类型 | 依赖的数据 |
|----------|-----------|
| **借贷协议** | 资产价格 → 计算抵押率和清算线 |
| **衍生品协议** | 标的资产价格 → 结算 |
| **保险协议** | 事件是否发生 → 理赔触发 |
| **RWA（真实世界资产）** | 链下资产状态 → 代币化 |

## 1. Price Feed（价格喂价）

### 核心定义

**Price Feed** 是最常见的 Oracle 形式，为合约提供**资产价格**。

### 价格喂价的用途

| 用途 | 说明 |
|------|------|
| **抵押率计算** | 抵押品价值 / 借款金额 |
| **清算线判断** | 抵押品价值跌到多少触发清算 |
| **Swap 限价** | 确保用户不会以极端价格成交 |
| **借款额度** | 根据抵押品价值决定可借金额 |
| **资产净值** | 协议/金库的总体估值 |

### 一个价格数字背后的东西

价格看起来只是一个数字，但它背后包含：

```
$3000 = 1 ETH 的价格

背后依赖:
  ├── 数据源: 从哪些交易所/市场获取 (Binance, Coinbase, Uniswap...)
  ├── 更新精度: 每秒更新 vs 每分钟 vs 每小时
  ├── 聚合方式: 中位数、加权平均、剔除异常值
  ├── 异常处理: 某个源挂了怎么办？价格偏离多大算异常？
  └── 延迟容忍: 多久不更新算是过期/stale
```

## 2. Data Feed（数据喂价）

### 核心定义

**Data Feed** 不限于价格，可以包含任何链外信息。

### Data Feed 可以提供的链外数据

| 类型 | 示例 |
|------|------|
| **储备证明** | 某稳定币发行方的银行账户余额 |
| **利率** | 国债收益率、Libor |
| **宏观数据** | CPI、GDP、汇率 |
| **投票结果** | DAO 链下投票、治理结果 |
| **天气** | 参数型保险赔付触发 |
| **随机数** | 可验证随机函数 (VRF) 生成的随机数 |
| **体育比分** | 预测市场结算 |

### 任何依赖外部数据的合约都要回答同一组问题

```
问题清单:
  1. 数据从哪里来？        → 来源可信吗？
  2. 如何更新？            → 推送还是拉取？
  3. 谁可以提交？          → 中心化还是去中心化？
  4. 是否可以验证？        → 能追溯到原始数据源吗？
  5. 出错后怎么处理？      → 有回滚/暂停机制吗？
```

## 3. Oracle Risk（预言机风险）

### 核心定义

**Oracle Risk** 是链外数据进入链上执行时引入的**系统性风险**。Oracle 错误不只是"显示错误"——它会直接触发清算、借贷、结算等真实的资产变动。

### 常见风险清单

#### 3.1 数据源被操纵

```
攻击者在低流动性交易所制造极端价格
      ↓
Oracle 聚合了这个价格
      ↓
链上借款协议以操纵后的价格清算用户
      ↓
攻击者以低价买入被清算的抵押品
```

#### 3.2 Feed 更新延迟

```
市场剧烈波动: ETH 从 $3000 急跌到 $2000
Oracle 延迟: 还显示 $3000

结果: 应该被清算的仓位没有被清算 → 坏账积累
```

#### 3.3 聚合节点离线

```
多个价格数据源中部分节点掉线
聚合结果基于过少的数据源 → 不够可靠
```

#### 3.4 小数位/单位理解错误

```
Oracle 返回价格: 300000000000 (18 位精度)
合约按 300000000000 处理 → 价格被放大 10^18 倍

真实含义: 3000.000000000000000000
```

#### 3.5 低流动性资产价格被攻击

```
小市值代币只在 1-2 个 DEX 有流动性
攻击者用闪电贷短暂拉升价格
Oracle 取了这个价格 → 借贷协议被榨干
```

#### 3.6 合约没有检查 Stale Price（过期价格）

```
Oracle 最后一笔更新是 2 小时前
合约直接用这个 2 小时前的价格做清算判断
市场已经变了 → 清算决策是错误的
```

#### 3.7 Oracle 升级或权限变更不透明

```
预言机合约升级逻辑或被多签控制
如果权限在少数人手里 → 他们可以篡改价格
如果升级不通知用户 → 用户不知道自己依赖的数据源变了
```

## 4. AI Oracle

### 核心定义

**AI Oracle** 是把模型推理、评分、分类或生成结果提交给链上系统的设想和实践方向。

### 与传统 Oracle 的区别

| | 传统 Oracle | AI Oracle |
|------|------|------|
| 输入 | 交易所价格、API 数据 | 链上数据 + 链外上下文 |
| 输出 | 一个价格（数字） | 评分、分类、生成文本、决策 |
| 可验证性 | 相对容易（对比多个数据源） | 困难（模型输出难以直接验证） |
| 延迟 | 低延迟更新 | 模型推理需要时间 |

### AI Oracle 的潜在场景

| 场景 | AI Oracle 做什么 |
|------|-----------------|
| **信用评分** | 分析链上行为，给地址打分 |
| **内容审核** | 判断 UGC 内容是否符合规则 |
| **智能风控** | 识别可疑交易模式 |
| **市场分析** | 生成链上市场报告 |
| **治理辅助** | 总结提案、标记风险 |

### AI Oracle 的特殊挑战

```
1. 模型输出的不确定性 → 两次推理可能不同
2. 无法直接验证 → 需要额外的验证机制
3. 模型偏见 → 训练数据和模型选择影响结果
4. 推理成本 → 链上验证 ML 推理的计算开销
5. 透明度 → 模型权重和逻辑是否公开
```

## Oracle 全景总结

```
链外世界                           Oracle                          链上合约
┌──────────┐                    ┌──────────┐                    ┌──────────┐
│ 交易所价格 │ ──── Price Feed ──→│          │                    │ 借贷协议   │
│ 银行储备  │ ──── Data Feed ───→│  Oracle  │ ──── 链上数据 ──→ │ 衍生品    │
│ 利率/CPI  │                    │  网络    │                    │ 保险协议   │
│ AI 推理   │ ──── AI Oracle ──→│          │                    │ RWA      │
└──────────┘                    └──────────┘                    └──────────┘

关键认知:
  数据源 = 风险点
  "价格只是一个数字" ← 但这数字背后是整个 Oracle 基础设施

安全检查清单:
  ✅ 数据源分散吗？
  ✅ 更新够快吗？
  ✅ 有聚合异常处理吗？
  ✅ 合约检查了 stale price 吗？
  ✅ 权限变更透明吗？
  ✅ 低流动性资产有额外保护吗？
```
