## 6. Smart Contract（智能合约）

### 核心定义

智能合约是**部署在区块链上的可执行规则代码**。一旦部署，所有人都可以公开查看和验证，但代码逻辑不可篡改。

**关键特性**：
- **公开透明**：代码开源，任何人可审计
- **自动执行**：满足条件自动触发，无需中介
- **不可篡改**：部署后代码不可更改（除非有升级机制）
- **去信任**：规则由代码执行，不依赖第三方

## 6.1 Solidity

### 定义

**Solidity** 是 EVM（以太坊虚拟机）生态中最常见的智能合约编程语言。语法类似 JavaScript，但运行环境完全不同。

### 与普通编程语言的区别

| 特性 | 普通编程语言 | Solidity |
|------|-------------|----------|
| **运行环境** | 本地服务器/电脑 | 区块链虚拟机（EVM） |
| **状态存储** | 数据库/文件 | 链上 Storage（永久、昂贵） |
| **执行成本** | 几乎免费 | 需要 Gas 费 |
| **错误处理** | 可回滚/修复 | 交易失败则回滚，Gas 不退 |
| **可见性** | 通常私有 | 完全公开 |

### 核心概念

| 概念 | 说明 | 示例 |
|------|------|------|
| **Storage** | 永久存储在链上的数据 | 合约状态变量 |
| **msg** | 当前交易的消息对象 | `msg.sender`（调用者地址） |
| **send/transfer** | 发送 ETH 的方式 | 向地址转账 |
| **modifier** | 函数修饰器，可复用的前置条件检查 | `onlyOwner` |
| **event** | 事件，用于记录和通知 | `emit Transfer(...)` |
| **external call** | 调用其他合约 | `otherContract.function()` |
| **revert** | 回滚交易，撤销状态变更 | `require(condition, "error")` |
| **权限控制** | 限制谁可以执行某些操作 | `onlyOwner`、`onlyAdmin` |

### 常用库

| 库 | 用途 | 说明 |
|----|------|------|
| **OpenZeppelin** | 安全合约标准库 | 提供 ERC20、ERC721、权限控制等标准实现 |
| **ERC20** | 代币标准 | 可替代代币（如 USDT、UNI） |
| **ERC721** | NFT 标准 | 非同质化代币（如 CryptoPunks） |
| **ERC1155** | 多代币标准 | 同时支持可替代和不可替代代币 |

## 6.2 EVM（以太坊虚拟机）

### 定义

**EVM** 是智能合约的执行环境，是以太坊网络的"操作系统"。

### 核心功能

| 功能 | 说明 |
|------|------|
| **代码部署** | 将编译后的字节码部署到链上 |
| **合约调用** | 执行合约函数，改变链上状态 |
| **Gas 计费** | 计算每个操作的成本 |
| **状态隔离** | 每个合约有独立的存储空间 |

### EVM 执行流程

```
Solidity 代码
    ↓ 编译
EVM 字节码（Bytecode）
    ↓ 部署
链上合约地址
    ↓ 调用
EVM 逐条执行字节码指令
    ↓ 完成
链上状态更新（或回滚）
```

## 6.3 ABI（应用二进制接口）

### 定义

**ABI** 是应用程序与智能合约沟通的接口规范。前端通过 ABI 理解合约的函数、参数、返回值和事件。

### ABI 包含的信息

```json
[
  {
    "name": "transfer",
    "type": "function",
    "inputs": [
      {"name": "to", "type": "address"},
      {"name": "amount", "type": "uint256"}
    ],
    "outputs": [{"name": "", "type": "bool"}]
  },
  {
    "name": "Transfer",
    "type": "event",
    "inputs": [
      {"name": "from", "type": "address", "indexed": true},
      {"name": "to", "type": "address", "indexed": true},
      {"name": "value", "type": "uint256"}
    ]
  }
]
```

### ABI 的作用

| 场景 | 说明 |
|------|------|
| **前端调用** | 通过 ABI 编码函数调用数据 |
| **合约交互** | 知道有哪些函数、需要什么参数 |
| **事件监听** | 知道有哪些事件、如何解析日志 |
| **类型安全** | 确保参数类型正确 |

## 6.4 Event（事件）

### 定义

**Event** 是合约向外部系统留下的**可索引日志**。事件存储在链上，可以被外部应用监听和查询。

### 特点

- **可索引**：可以通过事件参数进行过滤和搜索
- **低成本**：比 Storage 存储便宜很多
- **可监听**：前端可以实时监听事件并更新 UI
- **不可篡改**：一旦发出，永久记录在链上

### 使用场景

| 场景 | 示例 |
|------|------|
| **转账通知** | `emit Transfer(from, to, amount)` |
| **状态变更** | `emit Approval(owner, spender, amount)` |
| **操作记录** | `emit Vote(proposalId, voter, support)` |

### 事件监听流程

```
合约执行 → emit Event → 写入交易日志
                              ↓
                        前端/Web3 监听
                              ↓
                        更新 UI / 触发通知
```

## 6.5 Upgrade（合约升级）

### 为什么需要升级

- **安全漏洞**：发现 Bug 需要修复
- **功能迭代**：添加新功能
- **治理需求**：规则需要调整

### 升级的挑战

- 合约代码**默认不可更改**
- 需要预先设计升级机制
- 升级权限需要严格控制

### 常见升级方案

| 方案 | 原理 | 特点 |
|------|------|------|
| **代理模式（Proxy）** | 分离逻辑合约和存储合约 | 最常用，通过代理合约指向新逻辑 |
| **Diamond 标准（EIP-2535）** | 模块化合约，可替换部分功能 | 灵活，适合复杂合约 |
| **迁移模式** | 部署新合约，迁移数据 | 简单但成本高 |

### 权限控制

- **多签控制**：需要多个管理员同意才能升级
- **Ownable**：单所有者控制（风险较高）
- **DAO 治理**：通过社区投票决定升级

## 6.6 完整调用流程示例

### 场景：用户点击"投票"按钮

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   用户操作   │     │    前端     │     │    钱包     │
│  点击投票   │────→│ 读取钱包地址 │────→│ 展示交易信息 │
└─────────────┘     │ 和当前网络   │     │ 请求用户确认 │
                    └─────────────┘     └──────┬──────┘
                                                  │
                    ┌─────────────┐              │
                    │   根据 ABI   │←─────────────┘
                    │ 编码调用数据  │    用户签名
                    └──────┬──────┘
                           │
                    ┌──────▼──────┐
                    │  广播到 RPC  │
                    │   节点       │
                    └──────┬──────┘
                           │
                    ┌──────▼──────┐
                    │  验证者打包  │
                    │  进区块      │
                    └──────┬──────┘
                           │
                    ┌──────▼──────┐
                    │   EVM 执行   │
                    │  合约逻辑    │
                    └──────┬──────┘
                           │
                    ┌──────┴──────┐
                    │   成功/失败  │
                    │  发出 Event  │
                    │  或 Revert   │
                    └──────┬──────┘
                           │
                    ┌──────▼──────┐
                    │  前端监听    │
                    │  交易回执    │
                    │  更新页面    │
                    └─────────────┘
```

### 详细步骤

| 步骤 | 执行者 | 操作 | 结果 |
|------|--------|------|------|
| 1 | 用户 | 点击"投票"按钮 | 触发前端逻辑 |
| 2 | 前端 | 读取钱包地址和当前网络 | 确认用户环境 |
| 3 | 前端 | 根据 ABI 编码调用数据 | 生成交易数据 |
| 4 | 钱包 | 展示交易信息 | 用户看到详情 |
| 5 | 用户 | 确认并签名 | 授权交易 |
| 6 | 钱包 | 广播交易到 RPC 节点 | 交易进入网络 |
| 7 | 验证者 | 打包交易进区块 | 交易被确认 |
| 8 | EVM | 执行合约逻辑 | 状态变更 |
| 9 | 合约 | 成功则发出 Event | 记录日志 |
| 10 | 合约 | 失败则 Revert | 状态回滚 |
| 11 | 前端 | 监听交易回执 | 获取结果 |
| 12 | 前端 | 更新页面状态 | 显示投票结果 |

## 智能合约安全要点

| 风险 | 说明 | 防护 |
|------|------|------|
| **重入攻击** | 合约在转账时被递归调用 | 先更新状态再转账 |
| **整数溢出** | 计算结果超出范围 | 使用 SafeMath 库 |
| **权限漏洞** | 关键函数没有权限控制 | 使用 modifier 限制 |
| **前端依赖** | 过度信任前端输入 | 合约端验证所有输入 |
| **Gas 限制** | 循环操作消耗过多 Gas | 避免无界循环 |
