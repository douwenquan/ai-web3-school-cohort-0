# Web3 基础概念学习笔记：Indexing（数据索引）

> 基于 Handbook Web3 基础章节：Indexing  
> 把链上的海量原始数据变成可查询、可分析、可给 AI 用的数据层

---

## 0. 为什么需要索引层

### 原始链上数据的问题

```
一条链每天:
  ├── 数百万笔交易
  ├── 数十万个区块
  └── 无数合约 Event

直接通过 RPC 读取:
  ❌ 慢 —— 需要遍历区块
  ❌ 贵 —— 每次调用耗费 RPC 配额
  ❌ 难 —— 需要理解底层数据结构
  ❌ 不适合复杂查询 —— 没有 SQL/GraphQL
```

**索引层** 把这些原始数据整理成可查询的结构，让前端、分析和 AI 能高效访问。

---

## 1. Event Indexing（事件索引）

### 核心定义

**Event Indexing** 是监听合约日志，把链上的动作整理成**可以查询的记录**。

### 原始 Event → 可查询数据

```
合约发出:
  emit Transfer(from, to, amount)

索引器把它变成:
  ┌──────────┬──────────┬──────────┬────────┬────────────┐
  │   from   │    to    │  amount  │  tx    │ block_time │
  ├──────────┼──────────┼──────────┼────────┼────────────┤
  │ 0xabc... │ 0xdef... │  1000    │ 0x...  │ 1716...    │
  │ 0xdef... │ 0xghi... │   500    │ 0x...  │ 1716...    │
  └──────────┴──────────┴──────────┴────────┴────────────┘
```

### 常见的 Event 类型

| 合约类型 | Event 示例 | 索引后能查到什么 |
|----------|-----------|-----------------|
| **ERC-20** | `Transfer`, `Approval` | 所有转账和授权记录 |
| **DEX** | `Swap`, `Mint`, `Burn` | 交易历史、LP 操作 |
| **借贷** | `Deposit`, `Withdraw`, `Borrow` | 存借款流水 |
| **NFT** | `Transfer`, `Mint` | 铸造和转移历史 |
| **治理** | `VoteCast`, `ProposalCreated` | 投票记录 |

### 索引的价值

```
没有索引: "给我查一下这个地址的所有转账" 
         → 需要从创世区块开始扫 → 不可行

有了索引: "给我查一下这个地址的所有转账"
         → SELECT * FROM transfers WHERE from = '0xabc' → 秒级返回
```

## 2. Subgraph（子图）

### 核心定义

**Subgraph** 是 The Graph 协议中用**声明式方式**描述如何索引合约事件，并通过 **GraphQL** 暴露查询接口。

### Subgraph 的三个组成部分

```
┌─────────────────────────────────────────┐
│           Subgraph 定义文件              │
│                                         │
│  1. 要监听的合约和事件                   │
│     contract: UniswapV3Pool             │
│     events: Swap, Mint, Burn            │
│                                         │
│  2. Event → Entity 映射                 │
│     Swap event → SwapEntity             │
│     字段: sender, amount0, amount1, ... │
│                                         │
│  3. GraphQL Schema                      │
│     type Swap {                         │
│       id: ID!                           │
│       sender: String!                   │
│       amount0: BigInt!                  │
│     }                                   │
└─────────────────────────────────────────┘
```

### Subgraph 适合构建什么

| 场景 | 对应 Entity |
|------|------------|
| **代币** | Token, TokenHolder, Transfer |
| **DEX** | Pool, Swap, Position, Tick |
| **借贷** | Market, Deposit, Borrow, Liquidate |
| **治理** | Proposal, Vote, Delegate |
| **NFT** | Collection, Token, Trade |

### 查询示例

```graphql
# 查询某个地址的所有 Swap 记录
{
  swaps(
    where: { sender: "0xabc..." }
    orderBy: timestamp
    orderDirection: desc
    first: 20
  ) {
    id
    amount0
    amount1
    timestamp
  }
}
```

## 3. RPC（远程过程调用）

### 核心定义

**RPC** 是应用和区块链节点交互的接口，用来读取链状态、查询日志、估算 Gas 和发送交易。

### RPC 能做什么

| 操作 | 说明 |
|------|------|
| **读取链状态** | 查询余额、Nonce、合约存储 |
| **查询日志** | 获取合约 Event |
| **估算 Gas** | 交易前预估费用 |
| **发送交易** | 广播签名后的交易 |
| **调用合约** | 只读调用（不会产生交易） |

### ⚠️ RPC 的中心化风险

当前 Web3 生态的 RPC 供应商高度集中：

```
大部分 DApp → Infura / Alchemy / QuickNode
                    ↓
             少数几家供应商
                    ↓
         中心化风险:
           ├── 单点故障: 供应商挂了 → DApp 不可用
           ├── 审查风险: 供应商可过滤交易
           ├── 数据垄断: 少数实体掌握全网的访问入口
           └── 隐私问题: 供应商可记录所有请求
```

### 去中心化 RPC 的尝试

| 方案 | 说明 |
|------|------|
| **多 RPC 切换** | 前端配置多个 RPC 端点，一个挂了切另一个 |
| **去中心化 RPC 网络** | Pocket Network, Lava Network |
| **轻客户端** | 自己跑轻节点，不依赖第三方 RPC |

## 4. Data Pipeline（数据管道）

### 核心定义

**Data Pipeline** 是把链上数据、链下数据、索引结果和业务事件组合成一个**可以分析、可以监控、可以给 AI 使用**的完整数据系统。

### 数据管道的层级

```
                    ┌──────────────┐
                    │   应用层      │ ← AI Agent、仪表盘、通知
                    └──────┬───────┘
                    ┌──────▼───────┐
                    │   分析层      │ ← SQL 分析、ML 模型
                    └──────┬───────┘
                    ┌──────▼───────┐
                    │   索引层      │ ← Event Indexing, Subgraph
                    └──────┬───────┘
        ┌───────────────┼───────────────┐
        ▼               ▼               ▼
   ┌─────────┐    ┌─────────┐    ┌─────────┐
   │ 合约事件 │    │ RPC 数据 │    │ 链下数据 │
   └─────────┘    └─────────┘    └─────────┘
```

### 数据管道的价值

| 用途 | 说明 |
|------|------|
| **分析** | 协议 TVL、用户行为、市场趋势 |
| **监控** | 异常交易告警、清算监控、MEV 检测 |
| **AI 消费** | 给 Agent 提供链上上下文、风险评分 |
| **报表** | 投资组合、损益计算、税务报告 |

### AI × Indexing 的交叉点

```
AI Agent 需要链上数据来做决策 → 依赖索引层

示例: 交易风控 Agent
  1. 从索引层查询: 这个地址的历史清算记录
  2. 从索引层查询: 这个 DApp 的 TVL 变化趋势
  3. 结合 AI 推理: 生成风险评分
  4. 做出决策: 是否继续这笔交易
```

## Indexing 全景总结

```
合约
  │  emit Event
  ▼
┌─────────────────────┐
│   Event Indexing    │  监听日志 → 整理成可查询记录
└────────┬────────────┘
         │
         ▼
┌─────────────────────┐
│     Subgraph        │  声明式索引 + GraphQL 查询
│   (The Graph)       │  适合协议数据 API
└────────┬────────────┘
         │
         ▼
┌─────────────────────┐
│   Data Pipeline     │  链上 + 链下 + 索引 → 分析/监控/AI
└─────────────────────┘

基础设施层:
  RPC ← ⚠️ 中心化风险，少数供应商垄断访问入口
```
