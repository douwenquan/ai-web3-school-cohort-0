# AI 基础最小实践汇总

> 基于 Handbook AI 基础章节：LLM、Prompt、Context、RAG、Agent、Frameworks、VibeCoding、MCP、Evaluation、Fine-tuning、Inference

---

## 1. LLM — 交易解释器

### 目标
做一个"交易解释器"的最小版本。

### 输入
- 一笔交易哈希

### 输出要求
系统读取交易详情、事件日志和相关合约 ABI，然后让 LLM 生成一段解释，包含：
- 用户发起了什么动作
- 涉及哪些资产和地址
- 哪些信息来自链上数据，哪些是模型推断
- 模型不确定的地方
- 如果要签类似交易，用户应该检查什么

### 练习重点
不是让解释很漂亮，而是把 **模型生成、链上事实、来源边界、不确定性** 分开。

## 2. Prompt — 交易风险摘要

### 目标
写一个"交易风险摘要"prompt。

### 输入
- 交易目标地址
- 函数名
- 参数
- 资产变化
- simulation 结果
- 用户原始意图

### 输出格式（固定 JSON）
```json
{
  "summary": "",
  "asset_changes": [],
  "permissions_changed": [],
  "risk_level": "",
  "requires_human_approval": false,
  "uncertainties": [],
  "recommended_user_checks": []
}
```

### 测试用例
1. 普通转账
2. 无限授权
3. 目标地址与用户意图不匹配

### 验证点
模型是否能稳定标记风险和不确定性。

## 3. Context — 钱包授权检查 Agent

### 目标
为"钱包授权检查 Agent"设计一份 context spec。

### 场景
用户问："这个 dApp 要我 approve，可以签吗？"

### 必须拿到的上下文
- chain id 和当前区块
- token 合约、spender 地址、approve 数量
- 用户当前 allowance 和余额
- spender 是否在可信列表
- simulation 或静态检查结果
- dApp 页面提供的说明（标记为不可信外部内容）
- 用户本次意图

### 设计要求
- 哪些字段必须实时查询
- 哪些可以来自缓存
- 哪些不能被模型当成事实

## 4. RAG — 协议文档问答

### 目标
做一个"协议文档 RAG 问答"的最小版本。

### 实现步骤
1. 选择一个官方文档站
2. 抓取 10-20 篇页面
3. 按标题结构切 chunk
4. 存入向量库

### 测试问题（三类）
1. 文档中明确存在的 API 用法
2. 文档中不存在的问题，要求系统拒答
3. 旧版本或不确定版本的问题，要求系统提示需要核对版本

### 输出格式
```json
{
  "answer": "",
  "sources": [],
  "uncertainties": [],
  "needs_version_check": false
}
```

### 练习重点
**证据链**，不是 UI。

## 5. Agent — DAO 提案研究 Agent

### 目标
做一个"DAO 提案研究 Agent"的最小版本。

### 只读动作
- 读取提案正文
- 检索论坛讨论
- 总结支持和反对理由
- 标记缺失信息
- 输出投票前检查清单

### 限制
**不要让它直接投票**。

### 输出要求
明确说明：
- 用了哪些来源
- 哪些结论证据不足
- 是否发现治理或资金风险
- 如果用户要投票，还需要人工检查什么

### 进阶版本
设计权限升级：只有当用户明确授权，并经过投票交易 simulation 后，系统才允许生成投票交易草稿。

## 6. Frameworks — 框架对比

### 目标
做一个"文档问答 + 工具调用"的最小框架对比。

### 两种方式实现同一任务
1. **直接调用模型 API**：用户问题 + 检索结果 + JSON 输出
2. **使用框架**：同样的输入输出，但加入 tool schema、trace、失败重试和测试样本

### 对比维度
- 哪个版本更容易读懂？
- 哪个版本更容易加工具？
- 哪个版本更容易定位错误？
- 哪个版本更容易写回归测试？

### 练习重点
不是选冠军，而是**看清框架到底帮了你什么**。

## 7. VibeCoding — 完整工程闭环

### 目标
选择一个小功能，让 AI Coding Agent 完成一轮完整工程闭环。

### 步骤
1. 写清楚任务边界和不能改的文件
2. 让 Agent 搜索相关代码并给出计划
3. 让 Agent 做最小 patch
4. 运行测试或构建
5. 查看 git diff，手动审查结果
6. 要求 Agent 写一段 PR summary 和验证说明

### 记录要点
- 哪些信息给少了
- 哪些测试缺失
- 哪些改动需要人类判断

## 8. MCP — 只读 MCP Server

### 目标
做一个只读 MCP server。

### 暴露两个工具
1. `search_docs(query)`：搜索本地项目文档
2. `get_file(path)`：读取白名单目录里的文件

### 安全要求
- 不能读取白名单外路径
- 返回结果必须带来源路径
- 工具调用要写日志
- 错误要明确返回，而不是静默失败

### 进阶设计
"写入工具"的权限升级方案：
- 什么时候需要用户确认
- 如何撤销授权
- 如何审计每次调用

## 9. Evaluation — 最小评估集

### 目标
给一个"交易解释 / 文档问答 / Agent 工具调用"原型做最小 eval。

### 样本准备（30 条）
- 10 条正常问题
- 10 条边界或容易混淆的问题
- 5 条历史 bug 或预期失败样本
- 5 条恶意或注入样本

### 每条样本定义
- 输入
- 期望行为
- 必须包含的信息
- 必须拒绝或提醒的情况
- 是否需要引用来源

### 使用方式
每次改 prompt、模型或检索策略前后跑一遍，记录变化。

## 10. Fine-tuning — 微调前评估

### 目标
做一个"结构化摘要格式"的微调前评估。

### 先不要训练
准备 50 条样本：
- 输入：一段技术文档或提案
- 输出：固定 JSON 字段（summary、risks、open_questions、sources）

### 比较三种方案
1. 只改 prompt
2. Prompt + few-shot
3. 小规模 fine-tuning 或 adapter 方案

### 评估检查项
- 字段是否完整
- 是否编造来源
- 风险点是否漏掉
- 输出是否稳定

### 原则
只有前两种方案无法稳定解决问题时，再考虑 fine-tuning。

## 11. Inference — 推理服务对比

### 目标
做一个最小推理对比。

### 任务选择
例如：
- "总结一笔交易的风险"
- "从合约 ABI 中提取可调用方法"

### 对比方式
1. 用一个托管 API 模型跑一次
2. 用一个本地模型或较小模型跑一次

### 记录维度
- 延迟
- 成本
- 输出质量
- 隐私边界
- 失败情况

### 输出要求
写出你会在产品中选择哪一种，以及 fallback 怎么设计。

### 练习重点
不是证明哪个模型最好，而是**学会把模型选择放进真实约束里**。
