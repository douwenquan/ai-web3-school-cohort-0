## 5. Agent（智能体）

### 核心定义

Agent = **模型 + 工具 + Harness（编排框架）**

Agent 不只是"能聊天的模型"，而是能够**自主规划、调用工具、执行多步骤任务**的系统。它通过一系列的 Tool Use 调用外部功能，获取信息或使用外部工具来完成复杂目标。

### 关键机制

| 机制 | 说明 |
|------|------|
| **Plan 模式** | 将目标拆分为可执行的步骤，明确最终结果。Plan 需要包括：具体步骤、每步做什么、失败处理、验证方式 |
| **Reflection** | 检查中间结果，确认每个步骤的执行成果是否符合计划预期，并自动修正 |
| **Multi-Agent** | 将主 Agent 的任务拆分给多个子 Agent 执行，减少主 Agent 的 Context 占用。但需要确保拆分不会降低整体沟通效率 |

### 个人理解

- Agent 的核心价值是**从"回答问题"进入"执行流程"**
- Plan + Reflection 是 Agent 可靠性的关键，避免"一步错步步错"
- Multi-Agent 适合复杂任务，但通信开销是主要挑战

## 6. Frameworks（框架）

### 主流框架清单

| 框架 | 特点 | 适用场景 |
|------|------|----------|
| **LangChain** | 最早的 AI 应用框架，提供链式调用和工具集成 | 快速原型、工具链整合 |
| **LangGraph** | 基于图结构的 Agent 编排，支持循环和条件分支 | 复杂工作流、状态管理 |
| **OpenAI Agents SDK** | OpenAI 官方 Agent 框架，简洁易用 | OpenAI 模型生态 |
| **DSPy** | 声明式编程框架，优化 Prompt 和检索 | 检索增强、Prompt 优化 |
| **Hermes** | Nous Research 的 Agent 框架 | 研究型 Agent |
| **Learning Agent** | 学习辅助型 Agent | 学习记录、任务管理 |
| **Google Android Agent SDK** | Google 新推出的移动端 Agent SDK | 移动应用 |

### 个人理解

- 框架解决的是**编排问题**：如何组织模型、工具、上下文、记忆
- 不同框架适合不同阶段：快速原型用 LangChain，复杂工作流用 LangGraph
- 后续 Agent 开发时再深入了解具体框架

## 7. VibeCoding（氛围编程）

### 核心定义

通过自然语言与 AI 协作编程，让开发者专注于"做什么"而非"怎么做"。

### 常用工具

| 工具 | 类型 | 特点 |
|------|------|------|
| **Claude Code** | CLI | Anthropic 官方，深度集成 Claude 能力 |
| **Codex** | CLI + 桌面端 | OpenAI 官方，支持手机数据同步和远程操控 |
| **OpenCode** | CLI | 开源替代方案 |

### 当前局限

- **嵌入式开发**：AI 难以操作硬件设备验证代码效果
- **移动应用开发**：无法直接操作手机确认 UI/交互效果
- **需要人工验证**：复杂逻辑仍需开发者确认正确性

### 个人理解

- VibeCoding 是**效率工具**，不是替代开发者
- 适合：快速原型、重复代码、文档生成
- 不适合：硬件相关、复杂算法、安全关键代码

## 8. MCP（Model Context Protocol）

### 核心定义

MCP 是**模型上下文协议**，标准化模型与外部工具、数据源之间的连接方式。Agent 通过 MCP 访问和调用外部工具。

### 核心概念

| 概念 | 说明 |
|------|------|
| **Server** | 提供工具或数据的服务端 |
| **Client** | 调用 Server 的客户端（通常是 Agent） |
| **Tool Schema** | 工具的描述规范：名称、用途、参数、返回值、约束 |
| **Permission** | 权限控制，限制 Agent 能访问哪些工具 |

### 常用 MCP 工具

- **Context7** - 文档检索和代码理解
- **GitHub MCP** - 代码仓库操作

### Tool Schema 关键字段

```
- 工具名称
- 用途描述
- 参数定义（类型、必填、约束）
- 返回值说明
- 使用约束和限制
```

### 个人理解

- MCP 是 Agent 的"USB 接口"，标准化了工具连接方式
- Tool Schema 是核心，决定了 Agent 能否正确理解和使用工具
- 当前 MCP 生态还在发展中，部分实现可能不够成熟

## 9. 评估（Evaluation）与微调（Fine-tuning）

### 微调（Fine-tuning）

在预训练大模型基础上，使用特定数据集进行定向能力强化。

**常用方法：**

| 方法 | 说明 |
|------|------|
| **SFT（监督微调）** | 使用标注数据对模型进行监督学习 |
| **LoRA** | 低秩适配，只训练少量参数，高效微调 |
| **PEFT** | 参数高效微调，减少计算资源需求 |
| **Dataset** | 微调使用的训练数据集 |
| **Overfitting** | 过拟合风险：模型过度适应训练数据，泛化能力下降 |

### 评估（Evaluation）

微调后对模型进行系统评估，判断微调效果。

**评估维度：**
- 任务完成准确率
- 输出质量
- 与预期的对齐程度
- 泛化能力

### 个人理解

- 微调是**专业化和定制化**的手段，小模型在特定方向可以接近大模型
- 评估是微调的"质检环节"，不能省略
- 当前阶段**暂时不需要深入了解**，后续需要时再学习

## 10. 推理服务（Inference Services）

### 核心定义

AI 分为**训练**和**推理**两个阶段：

- **训练**：构建模型，调整参数（成本高，周期长）
- **推理**：将训练好的模型部署到服务器，给用户使用（日常调用）

### 部署方式

| 方式 | 说明 | 适用场景 |
|------|------|----------|
| **厂商 API** | 调用 OpenAI、Anthropic、Z.ai 等提供的 API | 快速接入、无需运维 |
| **本地部署** | 在自己的服务器上运行开源模型 | 数据隐私、成本控制 |

### 本地部署注意事项

- **参数规模**：大模型参数多，需要高性能硬件
- **精度降低**：为提升 TPM（每分钟令牌数），可能需要在部分环节降低精度
- **量化技术**：如 INT8、INT4 量化，减少显存占用

### 个人理解

- 推理是 AI 应用的"最后一公里"，直接影响用户体验
- 本地部署适合隐私敏感场景，但运维成本高
- API 调用简单，但依赖第三方服务稳定性

## 总结

| 概念 | 一句话总结 |
|------|-----------|
| Agent | 模型+工具+编排，能自主执行多步骤任务 |
| Framework | Agent 的"脚手架"，解决编排问题 |
| VibeCoding | 自然语言协作编程，效率工具 |
| MCP | 模型与外部工具的"USB 接口" |
| Fine-tuning | 定向强化模型能力，小模型也能专业 |
| Inference | 模型部署给用户使用，分 API 和本地 |
